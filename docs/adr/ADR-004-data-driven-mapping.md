# ADR-004: 五行到编程概念的映射采用可配置数据驱动

## 状态

已采纳

## 日期

2026-04-09

## 背景

DevFortune 的核心玩法是将传统五行生克理论映射到程序员的日常工作场景。例如：

- **金** -> 代码重构、架构设计（金主收敛、肃杀，对应精简代码）
- **木** -> 新功能开发、创意探索（木主生发）
- **水** -> 学习新技术、阅读文档（水主智慧、流动）
- **火** -> Code Review、团队协作（火主礼、光明）
- **土** -> 基础设施、运维部署（土主承载、稳定）

这些映射关系本质上是创意内容，没有"唯一正确"的对应方式。不同程序员对五行与编程概念的理解可能不同，社区也可能贡献更有趣的映射方案。如果把这些映射硬编码在 TypeScript 代码中，每次调整都需要修改源码、重新构建、发版。

## 决策

五行到编程概念的映射规则以 JSON 配置文件的形式存储在 `packages/core/data/mappings/` 目录下，与推算引擎代码分离。核心引擎提供默认映射文件，同时支持运行时加载自定义映射。

配置文件结构示例：

```json
{
  "$schema": "../schemas/mapping.schema.json",
  "version": "1.0.0",
  "id": "default",
  "name": "默认映射",
  "description": "DevFortune 默认的五行-编程映射方案",
  "mappings": {
    "metal": {
      "element": "金",
      "themes": ["重构", "架构设计", "代码审查", "性能优化"],
      "lucky": { "language": "Rust", "tool": "SonarQube", "activity": "删除冗余代码" },
      "unlucky": { "activity": "引入新依赖" }
    }
  }
}
```

核心引擎通过依赖注入方式接收映射配置，不直接读取文件系统（保持纯函数特性，与 ADR-002 一致）。文件读取由各端的适配层负责。

## 备选方案

### 方案 B：硬编码映射

- **优点**：实现最简单；类型安全有保证；无需解析和验证配置文件。
- **缺点**：每次调整映射内容都需要修改代码并重新发布；社区贡献门槛高（需要懂 TypeScript）；无法做 A/B 测试或用户自定义。

### 方案 C：插件系统

- **优点**：扩展性最强，社区可以发布独立的映射插件包。
- **缺点**：对当前项目阶段而言过度设计；插件加载机制引入额外复杂度；需要定义插件 API、版本兼容性策略等；在浏览器端动态加载插件有安全和性能问题。

## 后果

### 正面

- 非开发者也可以通过编辑 JSON 文件贡献新的映射方案，降低社区参与门槛。
- 支持多套映射方案并行存在，未来可以做 A/B 测试观察用户偏好。
- 映射内容的变更不需要重新编译代码，热更新成为可能（尤其对 Web 端）。
- JSON 配置文件可以被 JSON Schema 验证（见 ADR-005），保证结构正确性。
- 为未来的国际化（i18n）预留了空间，可以按语言提供不同映射文件。

### 负面

- 需要在运行时解析和验证 JSON，有少量性能开销（但映射数据量很小，可忽略）。
- 配置文件与代码分离后，重构时需要同时关注两侧，IDE 的自动重构无法覆盖 JSON 中的字符串。
- 需要额外维护 JSON Schema 和配置文档。

### 风险

- 自定义映射可能包含不当内容（冒犯性文案），如果未来支持社区共享映射方案，需要建立内容审核机制。
- JSON 配置的字段设计需要前瞻性，一旦发布后修改结构会涉及迁移问题。建议从 v1.0.0 开始使用 `version` 字段标记配置格式版本。
