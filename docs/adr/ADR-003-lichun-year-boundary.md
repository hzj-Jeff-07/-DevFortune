# ADR-003: 天干地支推算以立春为年界

## 状态

已采纳

## 日期

2026-04-09

## 背景

天干地支纪年系统中，"一年从哪一天开始"存在长期争议。主流存在两种观点：

1. **以正月初一（农历新年）为界**：民间习俗普遍采用此法，大众认知中"过了春节就是新的属相年"。各类生肖查询网站和日历 App 多采用此标准。
2. **以立春为界**：传统命理学（八字、紫微斗数等）严格以二十四节气中的"立春"作为年柱更替的分界点。立春是太阳到达黄经 315 度的时刻，时间精确到分钟。

两种标准的差异通常只影响每年 1 月下旬到 2 月中旬出生的人，但对于这些边界日期的用户，不同标准会给出完全不同的天干地支年柱，进而影响五行属性和运势结果。

DevFortune 作为一个带有命理色彩的趣味工具，需要在两种标准中做出选择。如果同时支持两种标准，会增加代码复杂度和用户困惑。

## 决策

采用**立春**作为天干地支年柱更替的分界点。具体规则如下：

- 立春时刻之前，仍使用上一年的天干地支年柱。
- 立春时刻及之后，切换为新一年的天干地支年柱。
- 立春的精确时刻采用天文算法计算（太阳黄经 315 度），精度达到分钟级别。
- 月柱同样以节气（立春、惊蛰、清明……）为界，而非农历初一。

在用户界面上，需要明确标注本项目采用的是"立春为界"的传统命理标准，并简要说明与民间习俗标准的区别，避免用户误解。

## 备选方案

### 方案 B：以正月初一（农历新年）为界

- **优点**：符合大众直觉，用户不会产生困惑；实现相对简单，只需要农历新年日期查表。
- **缺点**：与传统命理学的推算体系不一致；无法与主流命理工具的结果对照验证；如果用户对命理有一定了解，会质疑结果的专业性。

### 方案 C：同时支持两种标准，用户可切换

- **优点**：满足所有用户需求，体现技术中立。
- **缺点**：核心引擎需要维护两套推算逻辑；运势结果可能因标准不同而矛盾，造成用户困惑（"到底哪个准？"）；UI 上需要额外的切换控件和说明文案，增加认知负担。

## 后果

### 正面

- 与传统命理学体系保持一致，推算结果可以与权威命理资料对照验证。
- 立春时刻基于天文算法，有明确的数学定义，没有主观模糊地带。
- 月柱以节气为界的体系自洽，整个干支推算逻辑统一。
- 项目定位为"有文化底蕴的趣味工具"，选择专业标准增强了可信度。

### 负面

- 需要实现节气计算算法（VSOP87 行星理论的简化版本），开发成本高于简单查表。
- 部分用户可能不理解"为什么还没过春节就变了属相"，需要在 UI 上做解释说明。
- 立春精确时刻的计算涉及天文学，需要额外的测试验证。

### 风险

- 节气计算算法的精度如果不够，可能在立春当天出现年柱判断错误。建议与香港天文台或紫金山天文台发布的节气时刻表做交叉验证，覆盖 1900-2100 年。
- 需要处理时区问题：立春时刻应基于东八区（北京时间），无论用户所在时区。

## 参考

- 《协纪辨方书》——清代官方天文历法著作，明确以节气定年月
- Jean Meeus《天文算法》（Astronomical Algorithms）——节气计算的标准参考
- 中国科学院紫金山天文台发布的二十四节气时刻表
